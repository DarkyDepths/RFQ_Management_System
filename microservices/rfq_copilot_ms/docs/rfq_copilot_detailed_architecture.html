Session — entry point
RFQ-bound mode
when rfq_id is known at session start
  • rfq_id pre-loaded from session record
  • Profile + snapshot fetched once and stored in working memory
  • All subsequent turns use that cached context — no re-fetch unless stale
  • Role (estimation manager, engineer…) injected from IAM seam
Portfolio mode Refinement 1
when no rfq_id is bound — deterministic search first
  • Step 1 — deterministic: search_portfolio(query) runs against rfq_manager_ms
  • 0 results: LLM phrases a clarification — "I found nothing matching X, can you clarify?"
  • 1 result: auto-resolve, proceed as RFQ-bound
  • N results: LLM presents the list, engineer picks — LLM never guesses identity
  • LLM owns phrasing only, code owns identity resolution
Memory — two tiers, Phase 1
Working memory — short-term
scope: one session · lives in conversation DB
  • Last 6–10 conversation turns verbatim
  • Current rfq_id, user role, active stage
  • Pre-fetched RFQ profile and snapshot (RFQ-bound sessions)
  • Reset when session ends
  • Feeds context assembly on every turn
Episodic memory — long-term
scope: per rfq_id across sessions · append-only
  • Summaries of past sessions on the same RFQ
  • Key facts surfaced across multiple conversations
  • Turns older than the working memory window collapse here as compressed summary
  • Never deleted — appended only
  • Semantic and procedural memory deferred to Phase 2 (requires real usage data)
Context assembly — five ordered layers, target ~2 750 tokens
L1 — Prompt engineering
static · always present · ~800 tokens max
  • Role: RFQ Copilot for GHI / Albassam Group
  • Persona: industrial estimation expert, pressure vessels, Aramco standards
  • COT: think step-by-step internally before answering — never show thinking to engineer
  • Few-shot: 6 grounded Q→A examples covering hardest cases
  • Scope rules: what to answer, how to decline, what never to claim without a source_ref
  • Grounding rule: chatbot navigates and explains — never produces its own intelligence
L2 — State
injected per request from memory · ~300 tokens
  • Working memory: last N turns, rfq_id, role, stage
  • Episodic memory: past session facts for this RFQ if any exist
  • Pre-fetched profile + snapshot for RFQ-bound sessions
  • Older turns appear as episodic summary, not raw text
L3 — RAG
on demand only — not injected on every turn · ~400 tokens when called
  • ASME Sec VIII Div 1 & 2, TEMA, API 660/661
  • Aramco standards: SAES, SAEP, SAMSS, SATR
  • Fabrication norms: PWHT, NDE, RT, UT, weld procedures
  • GHI internal norms — protected, access-controlled
  • Max 3 passages retrieved, top-scored only — keeps tokens tight
  • Called via query_rag tool only when the question needs domain knowledge
L4 — Tool descriptions
always present · function signatures only · ~200 tokens
  • get_rfq_profile(rfq_id) → status, deadline, owner, priority, client, workflow
  • get_rfq_snapshot(rfq_id) → briefing, gaps, readiness, intelligence status
  • get_rfq_stages(rfq_id) → stage list, assigned team, blocker status
  • search_portfolio(query, filters) → matching RFQs across all sessions
  • query_rag(question) → passages from domain KB and GHI norms
L5 — Current message
appended last · ~50 tokens
  • What the engineer just typed
  • Placed last so it is the freshest signal in the context window
  • context_translator has already cleaned and validated it before injection
Main agent — one LLM call
COT reasoning — internal, never shown to engineer
chain-of-thought block · scratchpad only
  • Step 1: What exactly is being asked?
  • Step 2: Is this RFQ-bound or cross-portfolio?
  • Step 3: Is it in scope? If not — decline cleanly in one sentence and stop
  • Step 4: What evidence is needed to answer correctly?
  • Step 5: Which single tool to call, or none if context is already sufficient?
  • Step 6: What to claim and what never to claim without a source_ref?
Selective tool call
one call or zero — never all tools at once
  • Chosen by the LLM after COT reasoning — not pre-determined by code
  • Typed inputs only — no free-form strings passed to tools
  • Result returned as structured Pydantic object, flattened by context_translator before use
  • If no tool is needed (courtesy, follow-up, domain question already in RAG): answer directly, zero latency
Compose answer Refinement 3
chatbot boundary — navigate and explain only
  • Every RFQ-specific factual claim tied to a retrieved tool result
  • Industry-aware natural prose — not raw system state dumps
  • Gaps stated honestly when data is missing: "I do not have grounded data for X right now"
  • Never: invent gap analysis · generate compliance assessments · rewrite intelligence_ms briefings · guess what a workbook probably says
  • Intelligence lives in rfq_intelligence_ms — chatbot reads it, explains it, surfaces it
Tool layer — tool_executor dispatches, connectors call
get_rfq_profile
rfq_manager_ms
  • Input: rfq_id
  • Returns: status, deadline, owner, priority, client, workflow, progress
  • Use for: deadline, owner, status, priority, client questions
get_rfq_snapshot
rfq_intelligence_ms
  • Input: rfq_id
  • Returns: briefing, gaps, readiness state, intelligence status, review flags
  • Use for: "any gaps?", "what's the briefing?", "is this ready?" questions
get_rfq_stages
rfq_manager_ms
  • Input: rfq_id
  • Returns: stage list, current stage, assigned team, blocker status + reason code
  • Use for: stage, blocker, team assignment questions
search_portfolio
rfq_manager_ms — portfolio mode + cross-portfolio questions
  • Input: query string + optional filters (client, date range, status, stage)
  • Returns: ranked list of matching RFQs with key metadata
  • Use for: "last project with client X", "all RFQs in estimation stage", portfolio disambiguation
query_rag
Vector DB — domain knowledge + GHI internal norms
  • Input: natural language question
  • Returns: top 3 scored passages from domain KB
  • Use for: ASME, Aramco standards, PWHT, NDE, fabrication norm questions
  • Not called for RFQ-specific data — those go through profile / snapshot / stages tools
Deterministic guardrails — Phase 1 Refinement 2
Evidence check
every RFQ-specific claim must have a source_ref
  • Verifies that each factual claim in the answer is traceable to a tool result
  • Claims without a source_ref are stripped — gap stated honestly instead
  • Fast deterministic string check — no LLM call
Scope check
off-domain content blocked before reaching engineer
  • Detects answers that drifted outside RFQ management, estimation, or industrial domain
  • On failure: replace with one clean decline sentence + redirect
  • Deterministic pattern check on response content
Ambiguity check
Option C — one clarification, never a loop
  • Detects when the question cannot be answered without more information
  • On failure: ask exactly one clarification question, then stop
  • Example: "Do you mean the RFQ code or the internal UUID?"
  • Never loops — if clarification fails again, surface the gap honestly
Shape check
response format and length within bounds
  • Ensures response is within expected length range for the question type
  • Prevents wall-of-text dumps and one-word non-answers
  • Enforces that courtesy responses stay brief
LLM judge — Phase 2 Deferred
LLM-as-judge — second LLM call added after Phase 1 is proven
replaces or augments the deterministic guardrails above with semantic evaluation
  • Added only after the Phase 1 pipeline is stable and well-understood
  • Checks the same three things as Option C but with semantic reasoning, not pattern matching
  • Receives: user question + tool results used + answer draft — condensed, under 600 tokens
  • Verdict options: pass · fabricated fact detected · scope drift · ambiguous → each with a defined response path (Option C)
  • Never loops — on any failure, Option C applies: surface honestly, state what is grounded, ask one clarification if needed
Persistence and memory write-back
response_translator
formats the answer for the API response
  • LLM output string → TurnResponse API shape
  • Source refs attached to response payload
  • Confidence level noted (deterministic / pattern-based / absent)
Persist turn → working memory
closes the memory loop
  • Turn written to conversation DB after every request
  • Becomes part of working memory for the next turn in this session
  • When working memory window is full, oldest turns collapse into episodic summary
