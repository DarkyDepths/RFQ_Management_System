rfq_copilot_ms — v3 path-aware
Architecture v3 — One canonical pipeline, 8 path profiles
The architecture is a single deterministic pipeline. The Path Planner classifies every turn into one of 8 paths, and each path activates only the stages it needs.
Section 1 shows the canonical pipeline. Section 2 shows the execution trace per path. Section 3 lists invariants, guardrails, and triggers.

Not every turn executes every stage. The Path Planner emits a TurnExecutionPlan that activates or skips stages per path. Section 2 shows the per-path execution traces.

Section 3 — Quick reference
9 Invariants (apply globally)
  • I1. Context packet rule: only requested fields injected, never raw cached payloads.
  • I2. Resolver vs Evidence: Resolver tools off-agent, never in L4.
  • I3. Access ordering: Access policy runs before any evidence tool call.
  • I4. Existence-leak policy (default vs strict mode toggle).
  • I5. Escalation inheritance: caller's resolved state carries over via EscalationGate.
  • I6. Access ≠ evidence: access always via manager_ms regardless of evidence source.
  • I7. Cross-target isolation (Path 7): per-target labelled packets.
  • I8. Cross-context memory fan-out (Path 6): working from owner, episodic from current target.
  • I9. Aggregate-count leak prohibition: counts/sums/sizes of inaccessible items MUST NOT be exposed (Paths 3, 7).
Pipeline gates (named components)
  • Path Planner · deterministic classifier with tiny-LLM fallback when uncertain · emits TurnExecutionPlan (which stages are active for this turn)
  • Evidence Check · deterministic gate between Agent.tool_call and Agent.compose · empty/404/0-passages → escalate before composition (never let LLM fabricate)
  • Escalation Gate · single intercept point for all stage failures · consumes ESCALATION_TRIGGERS · applies I5 inheritance · routes to Path 8.x sub-case
7 Guardrails (per-path subset)
  • scope · drift detection, redirect
  • evidence · every claim has source_ref
  • shape · response length / format bounds
  • access · pre-retrieval permission check
  • ambiguity_loop_check · never re-ask same clarification
  • comparable_field_policy · A/B/C filter (Path 7)
  • target_isolation · cross-target leak prevention (Paths 6, 7)

  • pending_clarification · written by 8.3, read by Family 7 (clarification response)
  • last_search_results · written by Path 3, enables "show more" / "the 3rd one" follow-ups
Notation conventions
  • dynamic: ... · cell content depends on requested field/sub-topic (e.g. Path 7 Allowed evidence)
  • [default → END] · path is terminal — finalizer is the exit, no further escalation
  • Soft trigger · modifies finalizer template (e.g. staleness caveat) — path stays the same
  • Hard trigger · changes the active path via Escalation map (always routed through Escalation Gate, never back to Planner)
  • Resolver/Evidence categorization is per-path, not per-tool · same tool may be Resolver in one path and Evidence in another
  • Evidence Check fail escalates before compose · never let the LLM compose against missing evidence
