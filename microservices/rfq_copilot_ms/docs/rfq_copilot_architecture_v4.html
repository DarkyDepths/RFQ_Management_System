rfq_copilot_ms — v4 trust-boundary
Architecture v4 — Two intake sources, one plan factory, Path Registry
v4 supersedes v3. Every turn enters via one of two classification sources: a deterministic FastIntake (Stage 0, anchored full-match regex for greetings / thanks / farewells / empty / pure punctuation) or a GPT-4o Planner (everything else). Both feed a single ExecutionPlanFactory, the only code permitted to construct a TurnExecutionPlan (CI-enforced — §11.5). The PlannerValidator catches LLM-output structural failures only; all policy enforcement (allowed/forbidden fields, intent_topic existence, confidence threshold, Group C, query slots, alias normalization) lives in the factory. Failures route through a single Escalation Gate that re-enters the factory via build_from_escalation.
Frozen: 2026-04-27 · See docs/11-Architecture_Frozen_v2.md for the canonical specification this diagram renders.
Supersedes: rfq_copilot_architecture_v3.html. The "deterministic + tiny LLM fallback" planner and the "Agent" tool stage are explicitly forbidden by the freeze.
Trust-boundary mantra: Two classification sources. One plan factory. One Escalation Gate. The LLM produces language; code produces truth; policy enforces boundaries; the Judge verifies; templates render the safe answer when nothing else worked.
Section 1 — Trust boundary legend (read this first)
Every block in every diagram is colored by who owns the decision it represents. If you can name the color, you know who's responsible. If a block looks the wrong color, that's a bug in the architecture, not a styling issue.
Code (deterministic) Pure functions, hard rules. FastIntake (Stage 0 anchored regex), PlannerValidator (LLM-output structural checks only — no Path Registry reads), ExecutionPlanFactory (the single TurnExecutionPlan constructor), Resolver, Access, Memory Load, Tool Executor, Evidence Check, Context Builder, Guardrails, Finalizer, Persist.
LLM (GPT-4o, JSON-schema-enforced) Three roles only: Planner (classify intent — emits PlannerProposal), Compose (render answer), Judge (verify draft). Never picks tools, never decides access, never owns escalation. Runs only when FastIntake misses.
Path Registry (config) Single source of policy truth. Owns: allowed_intake_sources (which classification source may emit each intent), allowed/forbidden tools and fields, field aliases, confidence thresholds, guardrails, judge triggers, memory policy, persistence policy, finalizer template keys. Read at runtime only by ExecutionPlanFactory and Escalation Gate — CI-enforced (§11.5.2).
Escalation Gate (deterministic intercept) Single point that catches every stage's failure trigger and re-enters ExecutionPlanFactory.build_from_escalation() to construct the Path 8.x plan with a reason_code. Stages do not handle their own escalation. The Gate never instantiates TurnExecutionPlan directly.
External services Azure OpenAI (LLM backend), rfq_manager_ms (operational truth), rfq_intelligence_ms (derived truth), Domain KB / RAG (bounded knowledge).
Persistence threads, turns, audit_log, execution_records (forensics, partial-write, carries intake_path for FastIntake hit-rate analysis), session_state.
Section 2 — 8 Path overview (path-aware view)
v3 was readable because it gave you a per-path mental map. v4 keeps that — same 8 paths, redrawn with the freeze vocabulary: PlannerProposal → PlannerValidator → ValidatedPlannerProposal → ExecutionPlanFactory → TurnExecutionPlan, deterministic Tool Executor (never an Agent), Escalation Gate with reason_code at every failure edge. Path 1, 8.2, 8.3 can also enter via FastIntake (Stage 0) directly to the factory, skipping the Planner LLM call. Stages omitted from a path are skipped at runtime — the engine is one canonical pipeline (Section 3), each path activates only what it needs.
Two universal conventions for the per-path diagrams below (to keep them readable):
1. llm_unreachable can fire at the Planner, Compose, OR Judge stage in any path that uses them. Routes to 8.5 llm_unavailable. To avoid 3 redundant arrows per path, the per-path diagrams omit these — assume every path that calls an LLM has them implicitly.
2. Reclassification is Planner/Validator-only — never mid-pipeline. If the user's intent doesn't match the path being executed, that's caught by the Planner on the next turn (or by the Validator pre-execution), not by routing arrows out of mid-pipeline stages. The freeze forbids "user_asks_X mid-tool-execution → switch to Path Y" patterns.

Section 7 — Implementation order (vertical slices)
Each slice ships one path end-to-end through the full pipeline. No horizontal "build all planners then all resolvers." Slice 1 is the foundation — it ships the architecture itself, with Path 4 as the first answer path and Path 8 as the safety net.

1 Path 4 + Path 8 (8.1, 8.2, 8.3, 8.4, 8.5) + Path 1 trivial via FastIntake FastIntake (Slice 1 pattern table — greeting / thanks / farewell / empty / nonsense), Planner, PlannerValidator (LLM-failure checks only), Path Registry (with allowed_intake_sources declarations), ExecutionPlanFactory (the only TurnExecutionPlan constructor — implements rules F1..F8 + field-alias normalization), Resolver, Access, Memory Load, Tool Executor, Evidence Check, Context Builder, Compose, Guardrails (evidence + scope + shape), Judge, Finalizer with reason_code template selection, Escalation Gate (re-enters factory via build_from_escalation), execution_record with partial-write support and intake_path forensic field, ExecutionState, CI guards: single-construction TurnExecutionPlan, registry-reader allow-list, no-LLM-SDK in deterministic stages, FastIntake path-range guard. Path 4 is the demo-defining path. Path 8 is its safety net — they must ship together. Path 8.2 included so out-of-scope asks ("write me a recipe") are handled from day one. FastIntake + factory ship in Slice 1 because the trust-boundary architecture (single plan constructor, source-aware policy) only holds if every entry path obeys it from day one — bolting them on later means rewriting every stage's contract.
2 Path 1 (conversational) Reuses Planner, Validator, Finalizer. Tiny new template set. No fetch / no judge needed. Cheap, useful early, validates that the pipeline supports a path that doesn't fetch.

FastIntake (Stage 0) — Slice 1 pattern table
  • ^\s*$ → Path 8.3 empty_message
  • ^[^\w\s]+$ → Path 8.2 out_of_scope_nonsense (pure punctuation/symbols; word chars and digits never match — IF-0001 is correctly handed to the Planner)
  • ^(hi|hello|hey|salam|salut)[!.?\s]*$ → Path 1 greeting
  • ^(thanks|thank you|thx|merci)[!.?\s]*$ → Path 1 thanks
  • ^(bye|goodbye|cya|see you)[!.?\s]*$ → Path 1 farewell
  • Anchored full-match only — no fuzzy matching, no LLM, no DB
  • When in doubt, miss (false negatives cost a planner round-trip; false positives short-circuit a real question)
  • Path range: Path 1 / 8.2 / 8.3 only — CI guard §11.5.3 fails the build if a pattern emits any other path
ExecutionPlanFactory — rules F1..F8
  • F1. (path, intent_topic) exists in registry → else unsupported_intent_topic → 8.1
  • F2. IntakeSource ∈ IntentConfig.allowed_intake_sources → else intake_source_not_allowed → 8.1
  • F3. Field-alias normalization → canonical_requested_fields (unknown alias → unsupported_field_requested → 8.1)
  • F4. canonical_requested_fields ⊆ allowed_fields ∪ forbidden_fields → else unsupported_field_requested → 8.1
  • F5. canonical_requested_fields ∩ forbidden_fields == ∅ → else forbidden_field_requested → 8.1
  • F6. confidence ≥ IntentConfig.confidence_threshold → else confidence_below_threshold → 8.3
  • F7. Path 7: canonical_requested_fields ∩ comparable_field_groups.C == ∅ → else group_C_field_requested → 8.1
  • F8. Path 3: required_query_slots ⊆ proposal slot keys → else pre_search_query_underspecified → 8.3
  • Three input shapes: build_from_intake · build_from_planner · build_from_escalation
CI guards — mechanically enforced
  • §11.5.1 — Only execution_plan_factory.py may instantiate TurnExecutionPlan (AST grep)
  • §11.5.2 — Only execution_plan_factory.py + escalation_gate.py may import src.config.path_registry (import grep)
  • §11.5.3 — FastIntake patterns may only emit Path 1 / 8.2 / 8.3 (static check on pattern table)
  • §11.3 — Tool Executor may not import any LLM SDK (openai · anthropic · langchain · langgraph · llama_index · litellm · …)
  • All four guards run on every PR. Any red turns the build red.
Forbidden forever (by freeze)
  • ❌ LLM picks tools dynamically
  • ❌ LLM chooses guardrails
  • ❌ LLM decides what counts as evidence
  • ❌ LLM owns escalation routing
  • ❌ Stages handle their own escalation
  • ❌ Pipeline accepts PlannerProposal directly
  • ❌ Any code outside the factory constructs TurnExecutionPlan (CI §11.5.1)
  • ❌ Any code outside factory + gate imports the Path Registry (CI §11.5.2)
  • ❌ PlannerValidator reads the Path Registry — LLM-failure checks only
  • ❌ FastIntake fuzzy / substring / token-similarity matching, or LLM/DB/network — anchored full-match regex only
  • ❌ FastIntake emits a path/intent without IntakeSource.FAST_INTAKE (CI §11.5.3 + factory F2)
  • ❌ Planner emits Path 8.4 or 8.5 directly — those arise only from stage triggers
  • ❌ "Tiny LLM fallback" planner pattern
  • ❌ Confidence-only routing decisions
  • ❌ Free-text Planner / Judge output (must be JSON-schema-enforced)
  • ❌ Cross-target field mixing in Path 7
  • ❌ Silently downgrading Path 7 → Path 4
  • ❌ Treating manager-down / LLM-down / no-evidence as same user-facing message
  • ❌ Renaming "Tool Executor" back to "Agent" or letting it call an LLM
The four known risks (with mitigations)
  • Multi-turn context drift → bounded history (5 pairs), "answer only what's asked" rule, episodic summarization (later)
  • LLM-as-validator paradox (Judge) → deterministic Guardrails ALWAYS run before Judge; Judge is the last line, not the only line
  • Per-target leakage in Path 7 → per-target evidence packets, per-target labels, target_isolation guardrail, never silently downgrade
  • The "agreeable LLM" trap → strict system prompt rule, forbidden_inferences detector (Path 5), per-field "not recorded" rendering so absent fields can't be silently skipped
